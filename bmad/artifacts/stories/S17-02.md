# Story S17-02 — GitHub OAuth integration

**Epic:** Auth
**Sprint:** Sprint 17
**Points:** 3
**Priority:** Should

## User Story

As a **user**, I want to sign in with my GitHub account, so that I don't need to manage a separate password.

## Context

Better-Auth supports GitHub as a social provider. The PRD lists it as a "Should" requirement. The existing auth system uses email/password only.

## Acceptance Criteria

```gherkin
Given GITHUB_CLIENT_ID and GITHUB_CLIENT_SECRET are set
When a user clicks "Sign in with GitHub"
Then they are redirected to GitHub OAuth, then back to /app on success

Given the user has never signed in before
When GitHub OAuth completes
Then a new user record is created in the users table

Given the user already has an account linked to their GitHub email
When they sign in with GitHub
Then they are logged in to the existing account

Given GITHUB env vars are absent
When the sign-in page loads
Then the GitHub button is hidden
```

## Technical Notes

- Update `src/lib/server/auth.ts`: add `socialProviders: { github: { clientId, clientSecret } }` to Better-Auth config
- Update `src/routes/auth/+page.svelte`: add "Sign in with GitHub" button (conditional on env var)
- GitHub callback route handled automatically by Better-Auth at `/api/auth/callback/github`
- Env vars: `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET` (optional — button hidden when absent)
- Test: mock the OAuth flow with a stub provider

## Out of Scope

- GitHub avatar sync
- Unlinking GitHub from existing password account (deferred)

## Dependencies

- Better-Auth `^1.x` — check version supports `socialProviders.github`

## Definition of Done

- [ ] GitHub button visible when env vars set
- [ ] OAuth flow completes to /app in local dev
- [ ] Unit test covers config branching
- [ ] Docs updated if needed
